AI coding assistants and agents are now part of most teams' toolchains. Used well, they compress the boring parts of the job into seconds. Used badly, they inject subtle bugs faster than any human ever could, and they hide the reasoning that would let anyone else debug what shipped.

The difference between those two outcomes is not the model. It is the discipline of the person using it.

This post is about that discipline: which myths need to die, which facts have to stay in your head, why "blind pasting" is the single most damaging AI habit, and what an actually-safe workflow looks like when the assistant is doing most of the typing.

## Section 1: Myths and Facts

Every technology goes through a phase where the hype exceeds the reality. AI coding assistants are in that phase now. Four myths worth naming directly:

**Myth: AI writes production-ready code.**

Fact: AI writes plausible code. Production-readiness requires domain context, security awareness, and knowledge of your codebase's conventions. The model has none of that by default. What comes out looks correct because it is written in the shape of correct code. That is not the same as *being* correct.

**Myth: An agent that "just does the task" saves time.**

Fact: An agent amplifies whatever discipline you already have. On a codebase with tests, types, CI gates, and a clean architecture, agents are a genuine force multiplier. On a codebase without them, agents ship bugs at machine speed. The tooling around the agent is doing at least half of the work you attribute to the agent.

**Myth: If the code runs, it is correct.**

Fact: AI-generated code often passes the happy path and fails silently on edge cases the model did not consider. It happily returns `null` where an error should be thrown. It uses the first method it thought of, even when a better one exists in the standard library. It writes tests that only cover the code it just wrote, missing the cases that would have caught the bug in the first place.

**Myth: Using AI is itself a productivity metric.**

Fact: The productivity metric is *shipped correct behavior*. Time-to-generate is not the same as time-to-merge, and time-to-merge is not the same as time-to-no-incident. A team that generates 10x more code but debugs 3x longer is not more productive. It is more prolific, which is not the same thing.

None of these myths mean you should not use AI. They mean you should not confuse the tool with the discipline.

## Section 2: The Blind-Paste Antipattern

The single most damaging habit in AI-assisted development: pasting a generated snippet into a codebase without reading it line by line. It is fast, it feels productive, and it is how most AI-caused bugs enter production.

Four specific things break when you blind-paste:

**You inherit the snippet's assumptions.**

The model made choices about error handling style, retry policy, null-vs-undefined, sync-vs-async, whether to throw or return an error object. You never agreed to any of them. They are now in your codebase, silently, and future readers will assume you meant them.

**You pull in duplicate or conflicting dependencies.**

The snippet may `import lodash` when your codebase uses native ES methods. It may pull in `axios` when you standardized on `undici`. It may use `date-fns` when your project settled on `luxon` two years ago. Now your codebase has two libraries doing the same job, and nobody remembers which one is canonical.

**Your codebase's conventions get eroded silently.**

Your team names async functions with a `load` prefix, uses `snake_case` for database columns, throws typed errors instead of returning tuples. The snippet does none of that. After twenty pastes, the codebase looks like it was written by twenty different people, because effectively it was.

**Debugging becomes archaeology.**

Six months later, something breaks. You open the file. Nobody remembers writing this code. There are no comments, no commit message, no design doc. The person who pasted it did not read it either. You are now reverse-engineering behavior that no human ever actually intended.

The rule that fixes all four: **you may not commit code you cannot explain in a code review.**

Not "you should not." *You may not.* If you cannot narrate what every line does and why, either read it until you can, rewrite it in your own idiom, or reject the suggestion. The AI is a collaborator, not an author. Authorship still belongs to you, along with all the responsibility that word implies.

## Section 3: The Correct Workflow for AI-Assisted Code

The workflow that keeps AI useful without letting it degrade the codebase has six steps. Every one of them matters.

**1. Frame the task in writing before prompting.**

Inputs, outputs, edge cases, constraints. This alone eliminates most bad suggestions, because most bad suggestions come from ambiguous requirements. Writing the requirement forces you to know it. If you cannot write it, you are not ready to implement it, with or without AI.

**2. Prompt with your codebase's context loaded.**

File paths, existing helpers, naming conventions, error handling patterns. The more context you provide, the less generic the output. A prompt that says "use our existing `HttpClient` from `src/lib/http-client.ts` and follow the error pattern in `src/services/order-service.ts`" produces radically different output than "write a function that calls an API."

**3. Read every line the model produced.**

If any line is unclear, ask the model to explain it, then verify the explanation against docs or tests. Do not trust the model's explanation of its own code. It confabulates confidently and will happily justify code that does not do what it says.

**4. Rewrite in your own voice.**

Rename variables to match your codebase. Remove comments the model added. Adjust imports to match your style. Reorder the function to match your conventions. This is not busywork. It is the step that makes the code yours, and it is the step where you catch the assumptions that would have shipped invisibly.

**5. Write the test yourself.**

A model that generated the code cannot be trusted to write the test that validates the code. It will happily test that the code does what the code does, not that the code does what the requirement demands. That is a closed loop that catches nothing. The test is where you re-express the requirement independently, and where genuine mistakes surface.

**6. Attribute in the commit message if the change is non-trivial.**

Not for credit. For future debuggers who need to know that a human reviewed the reasoning, and which parts came from where. "AI-assisted; reviewed and adapted" is a useful signal in a git log two years from now.

## Section 4: Agents Specifically

An agent is an AI that can execute tools on its own: edit files, run commands, open pull requests, call APIs. Everything in the previous section still applies. But agents introduce new failure modes that plain code suggestions do not.

**Sandbox the blast radius.**

Agents should run in a branch, in a container, or against a fork. They should never have credentials that let them touch production. The default posture is "the agent can propose; a human commits."

**Gate everything through the same CI as humans.**

Agent-authored pull requests must pass the same lint, format, type, test, coverage, and security checks as human ones. No special path, no exceptions, no "well the agent said it works." If the checks are trustworthy for humans, they are trustworthy for agents. If they are not, the problem is the checks, not the agent.

**Require a human reviewer.**

Approvals from an agent do not count. This sounds obvious until you meet the team that configured an agent to auto-approve its own agent-authored PRs. The rubber-stamp reviewer is the failure mode you must actively design against.

**Log every action the agent took.**

File edits, terminal commands, network calls. When an agent goes sideways with no audit trail, you have a debugging problem that is significantly worse than debugging a human on the same task. The human at least remembers what they were trying to do. The agent's rationale, unless captured, is lost.

**Keep humans in the decision loop for anything with a blast radius.**

Deleting files, dropping tables, force-pushing, deploying, sending messages, spending money. These are actions where "the agent decided to" is not an acceptable postmortem sentence. If your agent config allows any of these without human confirmation, change it before the incident.

## Section 5: The Meta-Prompt for AI-Assisted Work

Every earlier post in this series shows a prompt structured to force the model to do real engineering. This one is different. It is the prompt for using the model itself.

**Weak (produces autopilot):**

> Can you write this function for me?

**Structured (produces assisted, not autopilot):**

> **Role:** Pair-programming assistant.
> **Context:** I am modifying [File] in a [Tech Stack] codebase. Existing helpers I want to reuse: [List]. Conventions I want to follow: [Naming, error handling, logging style].
> **Task:** Propose an implementation for [Requirement]. For each non-obvious line, explain why. Flag any assumption you made that I should verify.
> **Constraint:** Do not introduce new dependencies without listing them explicitly and stating the alternative I could write in-repo.

The model justifies itself, names its assumptions, and respects your existing code. It is slower than "write this function for me." That extra minute is where correctness lives.

What this prompt does not ask for: tests, design decisions, commits. Those stay with you. The model is the pair. It is not the driver, and if it is, the codebase has bigger problems than the model.

## Three Sentences to Repeat Before Every Commit

1. **Read every line before you commit it.** If you cannot explain it, you did not write it, and you cannot maintain it.
2. **Write the test yourself.** The model cannot validate its own output.
3. **Treat agents like junior engineers with root access.** Trust the reasoning, review the actions, log everything, keep humans in the loop for anything irreversible.

The teams that ship the best software over the next few years will not be the ones that use AI the most. They will be the ones that use it with the most discipline. Same tool, same day, radically different outcomes.
